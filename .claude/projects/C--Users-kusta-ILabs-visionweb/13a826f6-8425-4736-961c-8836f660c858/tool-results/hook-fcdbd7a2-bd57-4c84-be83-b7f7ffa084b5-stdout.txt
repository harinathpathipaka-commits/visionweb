DECISION INTELLIGENCE STORAGE\n\n### 7.1 Scoring Table Schema\n\nThe Decision Intelligence layer stores tuples of `(action, tool, context_type, scores)` and queries by context embedding similarity.\n\n```\nDecisionRecord {\n    id: Uuid,\n    action_hash: u64,          // hash of action description\n    tool_hash: u64,            // hash of tool name\n    context_embedding: [f32; 768],  // or 1536 depending on model\n    outcome_score: f32,        // 0.0 to 1.0\n    result_score: f32,         // 0.0 to 1.0\n    error_message: Option<String>,\n    error_penalty: f32,        // 0.0 to -1.0\n    business_outcome: f32,     // 0.0 to 1.0\n    composite_score: f32,      // weighted sum of above\n    timestamp: i64,            // Unix timestamp\n    count: u32,                // how many times this tuple fired\n}\n\n// Composite score formula:\n// composite = (outcome * 0.35) + (result * 0.30) + (error_penalty * 0.20) + (business * 0.15)\n// weighted by recency: recent outcomes count more\n```\n\n### 7.2 Storage Engine Decision: LanceDB (Rust-native embedded)\n\n**Why LanceDB:**\n- Written in Rust — can be embedded directly in the Rust process (no separate DB server)\n- Native vector search (cosine similarity, L2 distance) on `context_embedding`\n- Columnar storage (Apache Lance format) — efficient for the query pattern (read many rows, few columns)\n- Zero-copy reads via Arrow — integrates with the Arrow IPC system\n- Supports incremental writes — new scores appended without rebuilding\n- No external dependency — just a file on disk\n\n**Alternative considered: SQLite + sqlite-vec extension.** Viable and simpler, but sqlite-vec is newer and less battle-tested than LanceDB. LanceDB's columnar format is better for the vector-heavy workload.\n\n**Query pattern (Rust, via LanceDB):**\n\n```rust\n// Store after action completes\nlet record = DecisionRecord::new(action, tool, context_embedding, scores);\nlance_db.add(\"decision_records\", record);\n\n// Query when same context appears\nlet similar = lance_db\n    .search(\"decision_records\")\n    .nearest_to(&context_embedding, 5)?\n    .where_(\"composite_score > 0.5\")?\n    .order_by_desc(\"composite_score\")?\n    .execute()?;\n\n// Returns top-5 highest-scoring (action, tool) for this context\n```\n\n### 7.3 Python Access Path\n\nThe Decision Intelligence Engine (Python) accesses the storage through the Rust gRPC service:\n\n```\nPython DecisionEngine.score_action()\n    │\n    │ gRPC: StoreScore(record)\n    │──────────────────────────→ Rust: lance_db.add()\n    │\n    │ gRPC: QueryBestActions(embedding, k=5)\n    │──────────────────────────→ Rust: lance_db.search()\n    │←────────────────────────── ScoredActionList\n    │\n    Python receives top-k and uses LLM\n    to choose the best among them\n    (LLM can override if context is novel)\n```\n\n---\n\n## 8.
