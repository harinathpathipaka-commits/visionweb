MCP Gateway Proxy (`packages/mcp-server/`)\n\n**Entry Point** (`index.ts`): Creates a low-level MCP Server (stdlib transport) and registers handlers for `tools/list`, `tools/call`, `resources/*`, and `prompts/*`. Task context (taskType, customerId, agentId) is extracted from tool call `_meta` parameters or environment defaults.\n\n**Gateway Proxy** (`gateway-proxy.ts`): The core orchestration layer.\n- **tools/list**: Fetches raw tools from all healthy upstreams in parallel (`Promise.allSettled`), enriches descriptions based on mode (recommend/assist/auto), injects a virtual `li_recommend` tool in non-bootstrap modes, and sorts by ranking.\n- **tools/call**: Resolves mode, optionally fetches scores + policy decision for auto-mode rerouting (confidence threshold 0.3), proxies to upstream, fires-and-forgets outcome logging with LLM classification.\n- **Degradation**: Both list and call gracefully fall back to raw upstream passthrough when enrichment fails.\n\n**Tool Enrichment** (`tool-enrichment.ts`): Per-mode description formatting. Critically, `inputSchema` is NEVER modified -- only `description` is enriched. In recommend mode, success rates are woven in. In assist mode, ranked comparisons and directional warnings are added. In auto mode, descriptions are left unchanged (the agent never knows LI exists).\n\n**Mode Manager** (`mode-manager.ts`): Resolves the effective mode per task type. Checks per-task-type overrides first, falls back to global mode, caches resolution for 5 minutes. Bootstrap mode (= no mode set) means observe-only with zero enrichment.\n\n**Outcome Classifier** (`outcome-classifier.ts`): Uses GPT-4o-mini via OpenAI-compatible API to classify raw tool call results. Features: LRU cache with configurable TTL (default 5 min), in-flight request coalescing, 3-second timeout, structured JSON output with full validation, rule-based fallback when LLM is unavailable. Classification outputs: task_type (from 14-value taxonomy), context, success, error_message, result_summary, business_outcome (resolved/partial/failed/unknown), confidence. Confidence gating: business_outcome only attached at >= 0.7 confidence; below that, a `delayed` feedback signal is registered for future webhook resolution.\n\n**Fail-Open / Circuit Breaker** (`fail-open.ts`): Three-layer protection:\n- **Layer 1**: Outcome logging fails -> write to local disk queue (.li-queue/*.json) -> retry with exponential backoff\n- **Layer 2**: LI API unreachable -> fail open -> agent executes directly -> outcome queued to disk\n- **Layer 3**: Everything fails -> agent executes as if LI does not exist -> zero footprint\n- **Circuit breaker**: 3 consecutive upstream failures -> open for 60s -> half-open probe\n\n**Decision Tracker** (`decision-tracker.ts`): Generates lexicographically-sortable decision IDs (`dec_{ts12}_{agent4}_{rand4}`), tracks episodes, buffers decisions with 5-second flush interval, and writes a secondary disk queue (`decisions.jsonl`) for durability.\n\n**Upstream Registry** (`upstream-registry.ts`): Manages upstream MCP server connections with periodic health checks (default 30s), circuit breaker integration, and optimistic-start (assumes healthy until first check).\n\n**Rest Client** (`rest-client.ts`): Two clients:\n- **LiApiClient**: Communicates with the LI REST API. Features: retry with exponential backoff (max 3 attempts, 429/502/503/504 + TIMEOUT/NETWORK_ERROR), 15s timeout, X-API-Key auth.\n- **UpstreamMCPClient**: Proxies MCP JSON-RPC to upstream servers. Features: session caching (Mcp-Session-Id), concurrency gating (configurable, default 100), 30s timeout, non-JSON response handling.\n\n### 2.
