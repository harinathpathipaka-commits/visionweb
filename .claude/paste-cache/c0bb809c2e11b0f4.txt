# LayerInfinite MCP Server — Implementation Plan

## Goal

Any MCP-compatible platform (Claude Desktop, Cursor, Windsurf, Cline, OpenClaw, etc.) can integrate LayerInfinite by adding an MCP server config — **no SDK needed**.

## Background

LayerInfinite currently requires developers to install and integrate a Python or TypeScript SDK using decorators (`@li.action`). The industry has shifted toward MCP as a universal integration protocol. By shipping an MCP server that wraps the existing Hono REST API, we unlock every MCP-compatible platform without maintaining per-framework SDK integrations.

---

## Architecture

```mermaid
graph LR
    A[Claude Desktop / Cursor / Cline / etc.] -->|MCP Protocol| B["@layerinfinite/mcp-server"]
    B -->|HTTP + X-API-Key| C[Hono REST API]
    C --> D[Supabase / PostgreSQL]
```

> [!IMPORTANT]
> The MCP server is a **thin proxy** — it calls the REST API via HTTP. It does NOT connect to Supabase directly. This preserves auth, rate limiting, validation, and all business logic in the existing API layer.

### Key Decisions

| Decision | Choice |
|----------|--------|
| Package | `@layerinfinite/mcp-server` (new package in `packages/mcp-server/`) |
| MCP SDK | `@modelcontextprotocol/server` v2 (latest stable) |
| Transport | stdio (default) + Streamable HTTP (remote) |
| Auth | `LAYERINFINITE_API_KEY` env var → forwarded as `X-API-Key` header |
| Admin gating | Optional `LAYERINFINITE_ADMIN_KEY` → unlocks admin tools |
| Backend | HTTP calls to Hono REST API — never touches DB directly |
| Schema validation | Zod v4 (required by MCP SDK v2 Standard Schema) |

## Proposed Changes (Phase 1 — Full Implementation)

### REST Client

#### [NEW] [rest-client.ts](file:///c:/Users/kusta/Layerinfinite/Outcome/packages/mcp-server/src/rest-client.ts)

Thin HTTP client class that:
- Reads `LAYERINFINITE_API_KEY` and `LAYERINFINITE_BASE_URL` from config
- Sends all requests with `X-API-Key` header
- Has methods: `get(path, params)`, `post(path, body)`
- Handles HTTP errors → MCP error responses
- Supports optional `X-Admin-Key` header for admin endpoints
- Uses native `fetch()` (Node 18+)

#### [NEW] [config.ts](file:///c:/Users/kusta/Layerinfinite/Outcome/packages/mcp-server/src/config.ts)

```typescript
// Required
LAYERINFINITE_API_KEY    // Agent API key
LAYERINFINITE_BASE_URL   // Default: https://api.layerinfinite.app

// Optional
LAYERINFINITE_ADMIN_KEY  // Unlocks admin tools (register/toggle action)
```

## Open Questions

> [!IMPORTANT]
> **1. MCP SDK version:** The MCP TS SDK v2 (`@modelcontextprotocol/server`) is currently pre-alpha but shipping Q1 2026. The stable v1 uses `@modelcontextprotocol/sdk` with `McpServer` from `@modelcontextprotocol/sdk`. Should we target v1 (stable, widely deployed) or v2 (newer API, split packages)?
> 
> **Recommendation:** Target **v1** (`@modelcontextprotocol/sdk` ^1.29.0) for immediate compatibility with all existing MCP hosts. Upgrade to v2 once it stabilizes.

> [!IMPORTANT]
> **2. Base URL default:** The REST API is currently deployed on Railway/DigitalOcean. What should the default `LAYERINFINITE_BASE_URL` be? Options:
> - `https://api.layerinfinite.app` (production domain)
> - User must always specify it explicitly

> [!WARNING]
> **3. Admin tools exposure:** Registering admin tools (register_action, toggle_action, reinstate_agent) in an MCP server means any LLM with access could invoke them. The `LAYERINFINITE_ADMIN_KEY` gating mitigates this, but should we add an extra confirmation step or make admin tools opt-in only?
